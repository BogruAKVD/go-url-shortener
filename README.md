# go-url-shortener

HTTP-сервис сокращения ссылок на Go.

## Как запускать

In-memory:

```bash
go run ./cmd/server -storage=memory
```

PostgreSQL:

```bash
go run ./cmd/server \
  -storage=postgres \
  -db-host=localhost \
  -db-port=5432 \
  -db-user=postgres \
  -db-password=postgres \
  -db-name=url_shortener
```

При первом старте с PostgreSQL создается автоматически таблица `short_links`.

Docker:

```bash
docker build -t go-url-shortener .
docker run --rm -p 8080:8080 go-url-shortener -storage=memory
```

Docker Compose:

```bash
docker compose up --build
```


## Алгоритм генерации и почему он работает

Алфавит из 63 символов: `a-z`, `A-Z`, `0-9`, `_`. Длина короткой ссылки: 10 символов.

Число возможных кодов:

```text
63^10 ~= 9.85 * 10^17
```

Алгоритм генерации:

1. Берется строка `originalURL + ":" + salt`
2. Считается `sha256`
3. Из первых 10 байт хэша строится код: каждый байт берётся по модулю длины алфавита, затеи берётся знак с соответсвующим индексом
4. Если кода еще нет в хранилище, запись сохраняется
5. Если по коду уже лежит тот же URL, возвращается существующий short code
6. Если по коду лежит другой URL, соль увеличивается и попытка повторяется

Термин `соль` здесь используется не совсем строго в `криптографическом` смысле. В данном контексте под солью я имею в виду дополнительное значение, которое добавляется к исходной строке перед хэшированием и может меняться при повторной попытке.

Отдельная map `originalURL -> shortCode` не нужна, потому что алгоритм детерминирован: для одного и того же URL сервис проходит ту же последовательность значений `salt` и находит ту же запись через `shortCode -> originalURL`.

По памяти это лучше, чем держать две map:

- одна запись в текущем варианте хранится один раз
- при двух map индексация дублируется
- полный URL, как правило, длиннее short code, поэтому обратный индекс особенно дорог в памяти

Если считать коды равномерно распределенными по множеству размера `M = 63^10`, то по парадоксу дней рождения ожидаемое число коллизий порядка:

```text
n * (n - 1) / (2 * M)
```

- при `n = 10^6` ожидаемое число коллизий порядка `5 * 10^-7`
- при `n = 10^8` ожидаемое число коллизий порядка `5 * 10^-3`

То есть даже при больших `n` ожидаемое число коллизий остается малым.

Каждый байт SHA-256 принимает 256 значений, а алфавит состоит из 63 символов. Поскольку `256 % 63 = 4`, символы с индексами 0-3 алфавита встречаются в кодах примерно на 1.6% чаще остальных. Для задачи сокращения ссылок этот перекос несущественен, но и не учитывается в оценке ожидаемого числа колизий.

Также такое решение хуже подходит для сценария с удалением ссылок. После удаления становится сложнее аккуратно работать с уже использованными кодами и "пропусками" в последовательности попыток, поэтому текущий вариант лучше подходит для модели append-only.
